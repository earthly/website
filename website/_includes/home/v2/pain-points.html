Nobody likes their CI/CD
Traditional CI/CD has many pain points
Nightmare to Maintain
Slow Builds
Slower = More Vendor Profit
Poor Monorepo Support
  • CI/CD pipelines are challenging to maintain due to the inability to run locally and reliance on live testing.
  • Limited ability to reuse logic results in complexity and duplicated code across build scripts.
  • Traditional CI/CD platforms often require specific languages like YAML, Bash, or Groovy.
  • Achieving pipeline parallelization typically requires rearchitecting CI scripts, adding to maintenance challenges.
  • More than half of CI/CD run time is spent re-executing things that haven't changed.
  • Sandboxed CI environments provide consistency and security but sacrifice build performance.
  • Builds require dependency downloads, installations, and container image rebuilds every time.
  • Remote caching strategies are crude and limited, only benefiting very specific situations.
  • Slower builds result in more profit for vendors, discouraging innovation that makes builds faster.
  • The CI/CD industry's pay-by-the-minute model misaligns vendor incentives with user productivity.
  • CI/CD platforms are ill-equipped for today's era of monorepo and polyrepo code layouts.
  • Complex setups require heavy DIY scripting for marginal improvements, and are difficult to maintain.
